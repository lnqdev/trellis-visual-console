# Trellis 可视化管理项目

## 目标

建设一个独立仓库的本地可视化项目，集中浏览用户本机上多个 Trellis 项目的规范、任务和 Workflow 信息。

它不是 Trellis CLI 的替代品，也不是代码编辑器，而是一个只读内容中心，帮助用户快速回答：

- 我有哪些项目使用了 Trellis？
- 每个项目有哪些规范？
- 当前有哪些活动任务和历史任务？
- 某个任务的 PRD、设计和实施计划是什么？
- 当前项目处于哪个 Workflow 阶段？

## 已确认决策

- 产品是独立仓库，不并入当前 Trellis monorepo。
- 产品面向个人本机使用，不支持团队共享、远程访问、账号权限或云端同步。
- 运行形态是本地只读服务 + 浏览器 Web UI；首版不封装 Electron/Tauri 桌面安装包。
- 服务只绑定 `127.0.0.1`，不对局域网或公网开放。
- 首版完全只读，不创建、编辑、归档、删除或变更 Trellis 项目中的任何文件和状态。
- 项目发现支持两种方式：用户指定目录递归快速扫描，以及手动添加单个项目；快速扫描是主要方式。
- 页面区分焦点项目和历史项目：只有焦点项目实时监听，历史项目只保留摘要快照，不持续访问文件系统。
- 焦点项目集合跨应用重启保留；服务启动时重新索引焦点项目并恢复监听。
- 首版内容范围包括项目概览、Spec 浏览、活动/归档 Task 及其规划资料、Workflow 摘要。
- Workspace Journal、Mem、Channel/Worker、`.trellis/.runtime`、配置编辑和其他运维数据不纳入首版。
- 可视化项目不强制依赖 `@mindfoldhq/trellis-core`；文件扫描、Markdown 解析、任务文档读取、页面 DTO 和 SSE 由本地服务负责。只有在确有必要复用任务校验或阶段语义时，才选择性接入 Core。

## 功能需求

### 项目发现与注册

- 用户可以选择一个扫描根目录，递归查找包含有效 `.trellis/` 结构的项目。
- 扫描结果展示项目路径、名称、可用性和基础概览，用户可以批量加入项目列表。
- 用户可以手动添加单个项目目录；服务端必须校验目录结构后再登记。
- 项目路径、显示名称、焦点状态和索引时间保存在可视化应用自己的本机注册表，不写入被查看项目。
- 项目被移动、删除或失去权限时，保留历史记录并标记为不可用。

### 焦点项目与历史项目

- 焦点项目进入时先重新索引，再启动 `.trellis/spec/`、`.trellis/tasks/` 和必要配置文件的原生事件监听。
- 焦点项目收到文件变化后，通过 SSE 通知 Web UI 局部刷新。
- 项目移出焦点后立即停止监听，但保留最后一次成功索引的摘要快照。
- 历史项目不启动活动监听器，只展示摘要、最后索引时间和可能过期状态。
- 用户重新打开历史项目时，先执行一次刷新，再决定是否加入焦点集合。

### 内容浏览

- 项目概览展示项目路径、可用性、monorepo 包信息、Workflow 名称和当前阶段说明。
- Spec 视图保留 `.trellis/spec/` 的目录层级，支持按文件阅读 Markdown，并展示所属包或层级。
- Task 视图区分活动任务和归档任务，展示状态、阶段、标题、时间、负责人、包和关联信息。
- Task 详情按需读取并展示 PRD、设计、实施计划、研究文件和 JSONL 上下文清单。
- 页面内容可以追溯到源文件路径和最后索引时间。
- 可以提供“在文件管理器中显示”或“在外部编辑器中打开”的入口，但不提供应用内写操作。

### 只读与安全

- 服务只读取已登记项目范围内的 `.trellis/` 内容，不接受前端传入任意路径。
- 服务不得写入被查看项目的 JSON、JSONL、YAML 或 Markdown 文件。
- Markdown 渲染需禁用或清理不安全 HTML。
- 本机注册表和摘要快照放在操作系统的应用数据目录；删除它们不得影响任何 Trellis 项目。

## 验收标准

- [ ] 可以指定目录递归扫描并批量添加有效 Trellis 项目。
- [ ] 可以手动添加单个项目，并对无效目录给出明确原因。
- [ ] 重启应用后项目列表和焦点集合仍然存在。
- [ ] 历史项目没有活动文件监听器，但能展示摘要和最后索引时间。
- [ ] 焦点项目进入时完成一次刷新并开始监听，文件变化能通过 SSE 触发局部更新。
- [ ] 移出焦点后监听器释放，仍能查看最后摘要快照。
- [ ] 可以浏览 Spec 目录树和 Markdown 正文。
- [ ] 可以浏览活动/归档 Task，并阅读其规划资料。
- [ ] 项目概览显示 Workflow 名称和阶段说明。
- [ ] 源文件读取失败、权限不足、格式异常和项目失效都能在页面中明确展示。
- [ ] 浏览和刷新过程不会修改被查看项目的 `.trellis/` 内容。

## 技术约束与建议

- 推荐前端使用 React + TypeScript + Vite，本地服务使用 Node.js + TypeScript。
- 推荐使用 HTTP API + SSE；只读场景不需要 WebSocket 双向通信。
- 推荐使用操作系统原生事件封装；轮询只作为网络盘、容器挂载等场景的兼容性降级。
- 首版使用本机注册表 JSON 和可重建摘要快照，不引入数据库服务器。
- 源项目 `.trellis/` 是唯一事实来源；历史摘要只是缓存，不能反向覆盖源文件。
