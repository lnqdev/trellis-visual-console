# Trellis 可视化管理项目技术设计

## 当前推荐架构

采用已确认的“本地服务 + Web UI”本地优先架构：界面使用浏览器技术展示，但本地文件访问、扫描、解析和监听由运行在用户电脑上的 Node.js 服务负责。用户启动服务后，应用自动打开本机浏览器页面。

可视化项目作为独立仓库维护。它不把 `@mindfoldhq/trellis-core` 作为强制依赖：本地文件扫描、Spec Markdown 解析、任务文档读取、页面 DTO 格式化和 SSE 通知都由可视化项目本地服务负责。只有当需要复用 Trellis 已定义的任务记录校验或阶段推断时，才选择性依赖已发布的 Core SDK；正式运行不依赖 Trellis 源码目录存在。

```text
浏览器 Web UI
  │ HTTP：查询项目、规范、任务详情
  │ SSE：接收数据变更通知
  ▼
本地只读服务（127.0.0.1）
  ├─ 项目注册表
  ├─ 指定目录递归扫描器
  ├─ Trellis 数据解析器
  ├─ 文件变化监听器
  └─ 内存索引/缓存
       │
       ▼
多个本地项目的 .trellis/
```

## 为什么不是纯浏览器方案

普通 Web 页面不能任意扫描本机目录。浏览器的目录选择能力要求用户授权，权限持久化和浏览器兼容性存在差异，也缺少适合本产品的统一、可靠文件变化监听机制。

因此纯浏览器方案适合一次性读取用户选中的目录，但不适合长期管理多个项目并实时反映规范、任务文件的变化。

## 项目发现

### 快速扫描

1. 用户在本地服务提供的目录选择入口中指定扫描根目录。
2. 扫描器递归查找名为 `.trellis` 的目录。
3. 通过 `.trellis/config.yaml`、`.trellis/tasks/`、`.trellis/spec/` 等结构确认有效项目。
4. 返回候选项目，用户可批量加入项目注册表。
5. 扫描期间忽略 `.git`、`node_modules`、构建产物、隐藏缓存目录和无权限目录；避免跟随可能逃逸扫描根目录的符号链接。

### 手动添加

用户选择单个项目根目录，本地服务校验其 `.trellis/` 结构后加入项目注册表。

### 项目注册表

应用在用户本机保存项目根路径、显示名称、最近访问时间和可用状态。注册表不写入被查看项目，也不应提交到项目 Git 仓库。

注册表同时记录项目的展示状态：

- `history`：历史项目，只显示最后一次索引快照和 `lastIndexedAt`，没有活动监听器。
- `focus`：焦点项目，进入时立即重新索引，并为 Trellis 相关目录建立监听器。
- `unavailable`：项目路径不存在、权限不足或结构无效，保留记录但不监听。

焦点状态和项目注册表都保存在可视化应用自己的本地数据目录中，跨应用重启恢复；服务启动后依次校验并重新索引焦点项目，再建立监听器。应用关闭时所有监听器自然释放，不会在后台常驻。

“本机注册表”不是数据库服务器，也不是 Trellis 的配置文件，而是可视化应用自己的小型 JSON 配置，例如：

```json
{
  "projects": [
    {
      "id": "stable-project-id",
      "path": "/Users/me/work/demo",
      "label": "demo",
      "state": "focus",
      "lastIndexedAt": "2026-07-15T10:00:00.000Z"
    }
  ]
}
```

文件放在操作系统的应用数据目录中（例如 macOS 的 `~/Library/Application Support/...`、Linux 的 `~/.config/...`、Windows 的 `%APPDATA%/...`），不放进任何项目仓库。

注册表和快照缓存分工如下：

- 注册表：保存“有哪些项目、项目路径在哪里、哪些是焦点项目”。
- 摘要快照：保存项目概览、Spec 文件树、Task 摘要、Workflow 摘要、索引时间和错误信息，供历史项目列表展示。
- 源项目：`.trellis/` 仍是唯一事实来源；Markdown 正文和详细文件内容在焦点项目中按需从源项目读取。

这样用户即使清空应用注册表，也只会丢失可视化应用的项目清单和缓存，不会影响原项目。

## 数据读取与实时更新

### 初次加载

1. 本地服务读取项目注册表和历史快照，不主动访问所有历史项目目录。
2. 对焦点项目执行一次初始索引：项目元数据、Spec 文件树、活动/归档任务摘要。
3. 对历史项目展示最后快照、最后索引时间和可能过期标识。
4. Web UI 通过 HTTP API 按需读取列表和 Markdown 正文，避免启动时加载所有大文件。

### 文件变化监听

- 不监听所有已登记项目，更不监听项目的全部源码和构建目录；只为焦点项目监听首版展示路径：`.trellis/spec/`、`.trellis/tasks/`、`.trellis/config.yaml`。
- 监听器使用操作系统原生文件事件后端（macOS FSEvents、Linux inotify、Windows ReadDirectoryChangesW），默认关闭轮询。
- 首版可用 Chokidar 或 `@parcel/watcher` 封装跨平台差异；服务层保留 `FileWatcher` 接口，以便未来接入 Watchman 或平台专用实现。
- 文件创建、修改、删除事件先经过短时间防抖，再只重建受影响项目或文件的缓存。
- 本地服务通过 SSE 向界面发送项目级失效通知。
- Web UI 收到通知后重新查询受影响的列表或详情，避免在前端复制复杂文件合并逻辑。
- 文件监听失效或系统休眠恢复后，执行一次项目级重新校验，保证最终一致。

### 焦点项目生命周期

1. 历史项目被用户加入焦点时，服务端先校验路径并完成一次重新索引。
2. 索引成功后创建项目级监听器，向 SSE 连接发布 `project-focused` 和后续资源失效事件。
3. 用户移出焦点时，服务端停止并释放监听器，保留最后成功快照和时间戳。
4. 项目被删除、移动或权限失效时，监听器关闭，项目转为 `unavailable`。
5. 前端关闭某个详情页面不等于移出焦点；焦点是显式的用户项目状态，避免页面路由变化频繁启停监听器。

### 监听资源策略

- 快速扫描只在用户主动执行时发生，不把扫描根目录变成永久监听范围。
- 只有焦点项目监听 `.trellis` 下的 Spec/Task 目录；历史项目完全不保留活动监听器。
- 详细 Markdown 正文按需读取，不预加载所有文件内容。
- 事件处理采用防抖、批量失效和增量重建，避免一次保存触发多次页面刷新。
- 对网络盘、容器挂载或原生事件不可用的目录，允许降级为低频快照校验，并在 UI 标记“非实时”。

## Web UI 信息架构

- 左侧项目导航分为“焦点项目”和“历史项目”，项目行显示可用状态、实时监听状态和最后索引时间。
- 主区域先展示项目概览，再进入 Spec、Task、Workflow 三个只读视图。
- Spec 使用目录树 + Markdown 阅读器；Task 使用活动/归档分组 + 任务详情；Workflow 使用阶段摘要和源文件定位。
- 历史项目打开详情前显示“数据来自最后快照”；加入焦点后先刷新，刷新成功再切换为实时状态。

## 启动与运行

- 开发环境提供一个命令同时启动 Node 服务和 Vite Web UI，并自动打开浏览器。
- 生产模式由 Node 服务托管构建后的静态 Web 资源，用户只需启动一个本地服务。
- 服务退出时释放所有焦点项目监听器；应用数据目录中的注册表和摘要快照保留。

## 查询模型

首版 API 建议按只读资源划分：

- 项目列表与项目概览。
- Spec 目录树与 Markdown 正文。
- 活动任务、归档任务、任务摘要与任务文件清单。
- PRD、设计、实施计划、研究文档和 JSONL 上下文清单正文。
- 数据错误、缺失文件和不支持格式的诊断信息。

## 安全边界

- 本地服务默认只绑定 `127.0.0.1`，不对局域网或公网开放；首版不设计登录、团队成员、远程路径、云端同步或跨机器共享。
- API 只接受项目注册表中的路径标识，不接受前端传入任意绝对路径直接读取。
- 所有读取必须校验真实路径仍位于已登记项目的 `.trellis/` 内。
- 服务不提供写文件、执行命令、启动 Agent 或变更任务状态的接口。
- Markdown 渲染需要禁用或清理不安全 HTML。

## 备选方案

### Electron/Tauri 桌面应用

文件访问和目录选择最直接，也可以复用 Web UI，但安装包、签名、自动更新和跨平台发布成本更高。后续可以把相同的 Web UI 和本地服务封装为桌面应用。

### 纯浏览器应用

部署最轻，但本地目录权限、持续访问和文件监听能力不足，不推荐作为多项目长期管理方案。

## 技术栈建议

- Web UI：React + TypeScript + Vite。交互以目录树、列表、Markdown 阅读和局部刷新为主，Vite 启动快且与 Trellis 现有 TypeScript 生态一致。
- 本地服务：Node.js + TypeScript，优先使用轻量 HTTP 框架和 SSE，不引入数据库作为首版前置条件。
- 文件监听：优先使用操作系统原生事件的跨平台封装，并在服务层完成防抖、路径归一化和批量缓存失效；轮询只作为兼容性降级路径。
- 共享类型：独立项目服务端定义面向 UI 的 DTO；Core 仅作为可选的领域语义适配层，不让浏览器直接依赖 Node 文件系统或 Core 内部模块。

该方案需要维护前后端构建和本地服务启动流程；相比之下 Electron/Tauri 的文件访问更直接，但桌面分发成本更高。监听器的具体库应通过多项目基准测试决定，而不是默认引入 Watchman 这类常驻守护进程。

## 监听方案对比

| 方案 | 资源特征 | 适用范围 | 首版结论 |
| --- | --- | --- | --- |
| 焦点项目 + 原生事件封装（Chokidar / `@parcel/watcher`） | 只有用户关注项目占用监听资源；空闲时主要由 OS 维护事件 | 日常多项目管理 | 推荐默认方案 |
| Watchman | 对超大仓库、很多项目和事件合并更成熟；需要额外守护进程和安装依赖 | 大型 monorepo、几十到上百项目 | 预留适配，不作为首版前置依赖 |
| 定时轮询/快照哈希 | 实现简单，但持续产生磁盘 I/O 和 CPU，延迟取决于周期 | 网络盘、容器挂载、原生监听不可用 | 仅作降级方案 |
| 监听所有源码文件 | 事件量和目录句柄随仓库规模增长，展示需求并不需要 | 不适合本产品 | 明确禁止 |

## 已确定的首版技术方向

- Web UI 使用 React + TypeScript + Vite。
- 本地服务使用 Node.js + TypeScript。
- 服务与页面使用 HTTP API + SSE；不引入 WebSocket。
- 具体 HTTP 框架和文件监听库在实现阶段通过最小依赖和跨平台基准测试确定。
