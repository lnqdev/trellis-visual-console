# 第一性原理分析：是否必须使用 Trellis Core SDK

> 本文记录迁移前 Web MVP 的架构取舍；当前桌面架构见 `docs/planning/design.md`。

## Phase 0：公理

1. `.trellis/` 文件是被查看项目的事实来源；如果展示结果与文件不一致，产品就失去可信度。之所以是公理，是因为只读内容中心不能凭空创造另一份状态。
2. 浏览器无法稳定、长期地直接访问用户任意本地目录；本地服务必须拥有文件系统访问能力。之所以是公理，是因为目录扫描和持续监听都依赖操作系统文件能力。
3. SSE 只是把变化通知传给页面的传输方式，不负责发现或解释文件变化。之所以是公理，是因为没有事件源时传输协议无法产生正确数据。
4. 引入 SDK 只有在它提供了项目需要的稳定语义或减少不可接受的重复规则时才有价值。之所以是公理，是因为依赖本身不是产品目标，会增加版本和边界成本。

## Phase 1：问题本质

**核心问题：** 在本地建立一个只读索引和查询服务，把多个 Trellis 项目的 `.trellis/spec/`、`.trellis/tasks/` 内容稳定地转成 Web 页面，并能在文件变化后通知页面刷新。

**成功标准：** 用户可以添加项目、看到规范和任务详情；外部编辑器修改文件后，页面在无需手动重启服务的情况下收到更新；应用不修改源文件；展示数据可以追溯到具体文件路径。

## Phase 2：假设挑战

| 假设 | 为什么要质疑 | 依据 | 结论 |
| --- | --- | --- | --- |
| 必须使用 Core SDK | 当前需求主要是文件发现、Markdown 展示和任务文档阅读，不是复用全部领域服务 | 公理 4 | 修改：Core 不是强制依赖，只在需要任务语义时选择性使用 |
| Core SDK 负责扫描所有项目目录 | 现有 Core 主要提供 task/channel/mem 领域原语，并没有面向多项目的目录扫描入口 | 公理 1、2 | 丢弃：扫描器属于本地服务 |
| 浏览器可以直接完成递归扫描和实时监听 | 浏览器目录授权和监听能力受限，无法形成跨平台长期管理契约 | 公理 2 | 丢弃：由本地服务扫描和监听 |
| 只要使用 SSE 就能做到实时更新 | SSE 没有文件变化来源，也不负责增量解析 | 公理 3 | 修改：文件监听器产生事件，SSE 只负责通知 |
| 首版必须引入数据库保存全部项目数据 | 只读场景可以用本地注册表 + 内存索引，数据本身已在文件系统中持久化 | 公理 1、4 | 暂不引入：首版使用注册表文件和可重建缓存 |
| 前端可以直接理解 Trellis 的所有文件格式 | 会把 Node 文件系统、Markdown/JSONL 解析和容错规则泄漏到浏览器 | 公理 1、2 | 丢弃：服务端解析为稳定 DTO |

## Phase 3：具体事实

1. `.trellis/spec/` 的主要内容是目录和 Markdown 文件，展示时需要目录树与正文读取。
2. `.trellis/tasks/` 同时包含 `task.json`、PRD/设计/实施 Markdown、研究文件和 JSONL 上下文，展示时需要文件清单、状态摘要和正文按需读取。
3. 用户要求只读，因此服务端只需要“扫描、解析、索引、通知”，不需要写回业务项目。
4. 项目添加后才需要长期监听；扫描根目录本身不需要持续监听。

## Phase 4：推导方案

1. 事实 1 + 2 → 本地服务拥有文件解析器，输出项目、Spec、Task 三类 UI DTO；前端只负责展示。
2. 事实 3 → 服务端不设计写 API，源文件仍是唯一事实来源，缓存可以随时丢弃并重建。
3. 事实 4 → 添加项目时做递归扫描，登记后只监听已登记项目的 `.trellis/` 子路径。
4. 公理 3 → 监听器对变化做防抖和局部失效，SSE 发送失效通知，页面重新查询受影响资源。
5. 公理 4 → Core 仅在任务状态校验、阶段推断等存在明确语义复用收益时接入；Spec 解析和项目扫描不通过 Core 绕行。

## Phase 5：验证与压力测试

- 可追溯性：每个架构结论都能回溯到文件事实、本地访问约束、只读边界或 SDK 成本公理，验证通过。
- 完整性：扫描、解析、缓存、通知、展示和安全边界均有对应方案，验证通过。
- 诚实性：没有把 SSE 当作监听器，也没有把 Core 当作万能文件服务，验证通过。
- 预演失败：如果项目很大、文件权限变化或系统休眠导致监听丢事件，服务必须能标记项目状态并触发项目级重新扫描；因此“最终一致 + 可重建缓存”比维持复杂永久索引更适合首版。

## 结论

首版不需要强制使用 Core SDK。最小正确架构是“本地 Node 服务直接解析已登记项目的 `.trellis/` 文件 + React Web UI + 文件监听器 + SSE 通知”。Core 作为可选依赖，仅用于避免重复实现已经稳定且确有展示价值的 Trellis 领域语义。

## 补充分析：本地文件监听是否应覆盖所有项目文件

### 公理

1. 用户只需要看到 Trellis 内容变化，而不是整个代码仓库的变化；监听无关源码不会增加产品正确性。——需求边界决定了可观测集合。
2. 本地操作系统已经提供目录变化事件；在事件可用时，持续轮询不是必要条件。——原生事件是更低成本的变化来源。
3. 监听范围、缓存规模和事件处理策略共同决定资源消耗；不能只比较监听库名称。——资源成本来自系统边界而非单个 API。

### 问题本质与成功标准

问题不是“如何监听所有文件”，而是“如何以低资源代价让已登记项目的 Trellis 摘要和当前页面及时反映变化”。首版目标是：不监听源码目录；普通本地磁盘在文件变更后约 1 秒内通知页面；空闲时不持续扫描文件内容；监听异常时可重新校验。

### 被挑战的假设

| 假设 | 挑战 | 结论 |
| --- | --- | --- |
| 必须监听项目全部文件 | 页面不展示源码，监听源码只增加事件量 | 丢弃 |
| Node 监听一定很耗资源 | 原生事件后端通常由 OS 维护，成本主要随被监听目录数增长 | 修改：限制路径并使用原生后端 |
| SSE 能直接提供实时数据 | SSE 只是传输，不会发现文件变化 | 修改：监听器产生事件，SSE 传输失效通知 |
| 轮询比原生监听更可靠 | 轮询在大目录和多项目下持续产生 I/O；只在原生监听不可用时有价值 | 修改：作为降级路径 |
| Watchman 一定比 Node 库更好 | Watchman 对大型规模有优势，但引入守护进程和安装复杂度 | 修改：保留适配点，首版不强依赖 |

### 具体事实

1. 首版只关心 `.trellis/spec/`、`.trellis/tasks/` 和必要的 `config.yaml`。
2. 项目只有在用户快速扫描或手动添加后才进入长期管理集合。
3. Markdown 正文可以按需读取，任务和规范摘要可以增量重建。
4. macOS、Linux、Windows 都有原生目录事件机制，但网络盘和容器挂载可能需要轮询兜底。

### 推导方案

限定路径 + 原生事件监听 → 空闲时不读取文件内容，资源主要是少量目录句柄和摘要缓存。

防抖 + 批量失效 + 按需读取 → 编辑器一次保存产生多个事件时只触发一次项目级刷新。

监听器抽象 + 轮询降级 → 普通本地磁盘使用低成本原生事件，特殊文件系统仍能工作但明确标记为非实时。

### 压力测试

预演失败场景包括：用户登记上百个项目、系统从睡眠恢复、目录移动、网络挂载失去原生事件。对应措施是只监听 Trellis 子目录、限制缓存、重新校验项目状态、必要时暂停非活跃项目详细监听，并提供低频快照降级。

### 结论

首版不监听所有项目文件，而是监听已登记项目的 `.trellis/spec/`、`.trellis/tasks/` 和 `config.yaml`。优先使用 OS 原生事件封装；Watchman 只作为未来大规模适配方向，轮询只作为兼容性降级。资源消耗的主要控制手段是缩小监听范围和按需读取，而不是盲目更换 Node 库。

## 补充分析：焦点项目与历史项目模型

### 公理

1. 实时监听只有在用户需要当前数据时才产生产品价值；对不关注的历史项目保持实时不会改善当下体验。——实时性必须对应用户意图。
2. 项目注册信息与项目实时状态是两种不同生命周期；保存项目路径不等于必须持续读取项目。——注册表可以脱离监听器存在。
3. 用户仍需要识别历史项目，即使其数据不是实时的；因此停止监听不能等同于删除项目。——历史可见性是独立需求。

### 问题本质与成功标准

核心问题是让用户在保存大量项目历史的同时，只为当前关注的少量项目支付实时监听成本。成功标准是：历史项目零活动监听器；进入焦点时先刷新再监听；退出焦点时释放监听器；页面明确显示快照的新鲜度。

### 被挑战的假设

| 假设 | 挑战 | 依据 | 结论 |
| --- | --- | --- | --- |
| 所有已登记项目都必须实时 | 历史项目的数据变化通常与当前工作无关 | 公理 1 | 丢弃 |
| 项目列表只能展示实时数据 | 历史项目可以展示带时间戳的最后快照 | 公理 2、3 | 修改 |
| 打开详情页才算关注 | 路由切换会导致监听器频繁启停，用户可能同时关注多个项目 | 公理 1、2 | 修改：使用显式焦点集合 |
| 退出焦点后应删除缓存 | 删除会使历史列表失去摘要并增加下次打开成本 | 公理 3 | 丢弃：保留可重建快照 |
| 焦点只能有一个项目 | 用户可能并行关注多个项目，资源成本可由集合大小控制 | 公理 1、2 | 丢弃：允许小规模焦点集合 |

### 具体事实

1. 快速扫描和手动添加已经提供项目注册入口。
2. 首版是只读应用，历史快照不会覆盖源文件，也不成为新的业务事实来源。
3. 监听器可以按项目创建和销毁，SSE 事件可以携带 `projectId` 做局部失效。
4. 历史项目可以通过 `lastIndexedAt` 明确告知用户数据可能过期。

### 推导方案

注册表 + 最后快照 → 历史项目无需访问文件系统也可展示基本信息。

显式焦点集合 → 用户意图决定哪些项目实时，避免路由变化控制资源生命周期。

进入焦点先索引再监听 → 防止从陈旧快照直接切换到“实时”标签。

退出焦点停止监听并保留快照 → 同时满足资源释放和历史可见性。

### 压力测试与验证

如果用户把大量项目全部设为焦点，资源仍会增长，因此 UI 应显示焦点数量和监听状态，未来可增加软上限或资源提示。所有结论可回溯到用户意图、注册与实时生命周期分离、历史可见性三项事实；没有跳过项目不可用和快照过期场景。

### 结论

焦点项目模型比“所有登记项目持续监听”更合理。历史项目保存路径和最后快照，默认零监听；焦点项目进入时刷新并建立监听，退出时释放监听器。焦点应是显式的项目集合，而不是当前浏览器路由的副作用。

用户已确认焦点集合跨应用重启保留。该选择不改变“非焦点零监听”原则：应用关闭时监听器释放，下一次启动只恢复并重新索引上次的焦点项目。
