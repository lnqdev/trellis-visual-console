# Windows x64 原生交付设计

## 1. 设计范围

本任务不新增桌面架构。它在 Windows x64 实体机上验证并交付 `07-20-desktop-client` 已实现的 Tauri 2 应用，发现平台缺陷时只修改对应的 Rust Core、Tauri adapter、NSIS hook 或平台文档，并对共享行为执行必要回归。

## 2. 环境边界

- 目标系统为 Windows 10 22H2 或 Windows 11 x64。
- 构建目标固定为 `x86_64-pc-windows-msvc`，使用 MSVC 工具链原生构建。
- WebView2 已安装与缺失两种场景分别验证；缺失场景只允许安装器使用官方 bootstrapper 联网。
- 测试数据放在独立临时目录，至少包含中文路径、UNC 共享、28 项目/2 焦点性能集、版本 1 数据和路径攻击 fixture。
- 每组破坏性卸载或清理验证前记录源项目文件摘要与 Git 状态，操作后再次比对。

## 3. 产物与安装

Tauri 配置继续使用当前用户 NSIS 安装模式。安装器文件名明确标识 x64，应用名、图标和标识符与 macOS 保持一致。验收记录保存文件大小、SHA-256、Windows 版本、CPU、内存和 WebView2 版本。

卸载矩阵分两轮执行：

1. 默认保留数据：卸载后应用数据仍在，重装恢复项目列表和焦点状态。
2. 明确删除数据：二次确认后仅删除 `%APPDATA%\Trellis Visual Console`，源项目摘要和 Git 状态保持不变。

## 4. 平台功能矩阵

功能验收沿用 macOS 报告的信息架构，并追加 Windows 专项：

- 目录选择器取消、并发忙碌和中文路径。
- 受控外部打开本地目录、Spec/Task 文件和 HTTPS 链接。
- 原生 watcher 事件、轮询降级、网络盘/UNC 行为和关闭释放。
- 版本 1→2 数据迁移、损坏数据恢复和卸载保留策略。
- 单实例聚焦、无后台常驻、无本地端口和无 sidecar。

## 5. 网络与性能

先在联网环境完成缺失 WebView2 的安装器引导，再确认 WebView2 已就绪。随后断开网络执行启动、项目读取、文件更新、任务中心、正文、目录选择和退出流程，同时审计进程树的 TCP/UDP 连接。

性能使用 28 项目/2 焦点 fixture，记录冷启动到页面可操作、稳定 RSS、连续 5 分钟每 10 秒 CPU 样本和关闭退出时间。任何预算调整都必须附原始测量和用户批准。
