# 阶段六验证与交付技术设计

## 总体策略

阶段六不新增产品功能面板，也不把验证脚本永久放入仓库。验证由四类机制组成：

1. `/tmp` 临时 Trellis fixture 和应用数据目录，覆盖真实文件系统、HTTP、SSE 与生产进程。
2. `/tmp` 临时 TypeScript 验证程序，通过现有可注入 watcher factory 验证难以稳定制造的降级、队列和资源释放路径。
3. Playwright 验证最终用户路径、错误恢复和浏览器控制台。
4. 中文交付报告记录环境、命令、结果、指标、缺陷修复和平台覆盖边界。

仓库只保留必要的缺陷修复、规范更新和验证报告；临时 fixture 与验证程序在任务结束后删除。

## 验证拓扑

```text
/tmp/phase6-validation-<id>/
├── data/                    # 独立 registry.json / snapshots.json
├── projects/
│   ├── focus-a/            # 正常焦点项目
│   ├── history-b/          # 正常历史项目
│   ├── invalid-structure/  # 缺少必需入口
│   ├── bad-content/        # 坏 YAML / JSON / JSONL / UTF-8
│   ├── permission-denied/  # 临时 chmod 故障注入
│   └── benchmark-*/        # 多项目基线
└── evidence/               # 临时响应、事件和进程输出
```

所有故障注入只作用于临时 fixture。权限测试必须在清理前恢复权限，避免遗留不可删除目录。

## 验证分层

### 项目发现与内容解析

- 扫描正常、无效、重复和忽略目录项目。
- 登记前后比较注册表，证明扫描本身不持久化。
- 坏 YAML、task JSON、JSONL 和非法 UTF-8 只产生局部诊断，不阻塞其他项目。
- 路径攻击、符号链接和白名单外文档继续返回稳定 400/404。

### 焦点生命周期与监听

- 真实 Chokidar 验证焦点 Spec/Task/配置变化与历史项目零事件。
- 重启前登记多个焦点/历史项目；重启后比较焦点注册项、运行时状态和内部活动监听器计数。
- 项目删除、移动、结构破坏或权限失败后，项目进入 `unavailable`、监听器释放、最后成功快照保留。
- 受控 watcher 验证原生启动失败降级到轮询、运行期错误降级，以及轮询失败后的资源清理。

### HTTP、SSE 与 Web

- 生产服务使用独立端口和数据目录启动；验证首页、静态资源、健康接口、未知 API 和全部只读接口。
- SSE 捕获事件数量、资源类型和负载大小；断开后确认服务可正常退出。
- Playwright 验证不可用项目恢复、历史快照提示、焦点切换和控制台/网络错误。
- Markdown 原始 HTML、无效 URL 选择和中文错误继续回归。

### 生产退出与交付

- 发送 `SIGTERM`，等待 Fastify、SSE 和 watcher 清理完成。
- 退出后立即在同端口重新启动或绑定，证明端口和资源已释放。
- 自动打开浏览器成功路径做实测；失败容错做代码审查，当前实现没有可注入 opener，不为单次验证扩大生产抽象。

## 指标基线

报告使用固定字段，便于未来版本对比：

| 指标 | 采集方式 |
| --- | --- |
| 环境 | macOS、架构、Node、pnpm 版本 |
| 扫描项目数与耗时 | 单调时钟包围 `ProjectCatalog#scan` 或 HTTP 扫描请求 |
| 登记/索引耗时 | 单项目和批量登记请求耗时 |
| 焦点监听器数量 | `ProjectRealtimeManager#getActiveWatcherCount()` |
| 监听模式分布 | `listRuntimeStatuses()` 的 native/polling/stopped 数量 |
| 待处理变化数 | `pendingChanges` 汇总与峰值 |
| SSE 事件数/大小 | 捕获数据帧计数和字节数 |
| 注册表/快照大小 | 应用数据 JSON 文件字节数 |
| 进程内存 | 服务稳定后 RSS 与 heap 使用量 |
| 退出耗时 | `SIGTERM` 到进程退出的持续时间 |

本阶段建立基线，不设置缺少历史数据支撑的苛刻性能阈值；出现超时、崩溃、资源泄漏或明显非线性异常时视为缺陷。

## 跨平台审查

### macOS 实测

- Chokidar 原生文件事件、权限故障、系统外部打开、SIGTERM 和端口释放。

### Windows/Linux 平台中立审查

- `node:path`、`win32.isAbsolute`、路径分隔符归一化和应用数据目录分支。
- 不依赖 shell 特有路径拼接或硬编码 `/Users`。
- Chokidar 配置不跟随符号链接，轮询降级不依赖平台事件类型。
- 外部打开只把已验证路径交给 `open` 包，不拼接平台命令。

报告必须把上述审查与真实实机结果分开，不能使用“已通过 Windows/Linux 验证”的表述。

## 缺陷处理与回滚

- 验证发现的缺陷先在任务 PRD 验收项下记录证据，再做最小修复并重跑受影响场景。
- 路径、存储、监听和退出清理属于高风险文件，修改后必须执行全量检查和只读摘要对比。
- 若指标采集需要扩大公共 API 或 UI，优先改用临时内部验证程序；只有产品运行期确实需要时才更新共享合同。
- 所有生产代码修复可通过单独回滚阶段六提交恢复；验证报告保留失败事实和未覆盖平台说明。
