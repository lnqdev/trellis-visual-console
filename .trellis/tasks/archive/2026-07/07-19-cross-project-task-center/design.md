# 跨项目任务中心与搜索筛选技术设计

## 设计目标

在现有本机只读控制台中增加一个全局任务中心，通过单次元数据查询聚合多个 Trellis 项目的任务，并在浏览器内完成搜索、筛选、汇总和排序。功能继续以项目快照为事实来源，不读取任务正文、不增加持久化索引，也不改变单项目 Task 页面已有的任务树和文档阅读能力。

## 已确认产品决策

- 默认项目范围为焦点项目，可切换为全部项目。
- 全部项目范围中，焦点项目任务优先，历史项目任务明确显示快照状态和最后索引时间。
- 不可用项目不混入任务列表，只在汇总区域提供数量和诊断入口。
- 默认任务集合为活动任务，可切换为归档或全部任务。
- 首版只搜索项目和任务元数据，不搜索 PRD、设计、实施计划、研究资料或 JSONL 正文。
- 跨项目结果使用扁平工作列表；单项目页面继续负责完整任务树、进度、祖先导航和文档阅读。

## 方案选择

采用“服务端聚合快照元数据，前端本地搜索筛选”的方案。

不采用前端逐项目请求详情：该方案会产生与项目数量线性增长的请求，加载和错误状态分散，也会重复实现项目状态合并逻辑。

不采用服务端按每次筛选条件查询：当前产品是个人本机应用，任务元数据规模有限；每次输入或筛选都访问服务端会增加合同复杂度和交互延迟，却不能带来明显收益。

不采用持久化搜索索引：首版不搜索正文，现有快照已经包含所需字段，引入数据库、索引文件或迁移机制不符合最小实现原则。

## 总体架构

```text
ProjectCatalog 注册表与快照
        │
        ▼
ProjectApiService.listTaskCenter()
        │  只投影元数据，不读取源文件正文
        ▼
GET /api/tasks
        │  共享 Zod Schema 校验
        ▼
useTaskCenter
        │  项目范围、集合、搜索、筛选、汇总、排序
        ▼
TaskCenter 扁平工作列表
        │
        └─ 点击任务 → 现有项目 Task 页面
```

## 服务端设计

### 聚合接口

在现有项目只读 API 中增加 `GET /api/tasks`。路由继续依赖 `ProjectApiService`，不新增独立领域服务。

`ProjectApiService.listTaskCenter()` 执行以下流程：

1. 通过 `ProjectCatalog.listProjectData()` 一次读取全部已登记项目及其可空快照。
2. 复用现有项目列表投影逻辑生成项目状态、运行时监听状态、快照新鲜度和诊断数量。
3. 只对 `state !== "unavailable"` 且存在快照的项目合并活动与归档任务，并为每条任务补充 `projectId` 和任务集合；不可用项目即使保留旧快照也必须排除。
4. 在单项目全部任务范围内建立 `sourcePath -> title` 映射，为子任务解析 `parentTitle`；父引用缺失、循环或关系损坏时将 `parentTitle` 降级为 `null`，不能中断其他任务投影。
5. 不读取 Task 目录、Markdown 或 JSONL 正文；历史项目不需要临时正文读取授权。
6. 使用共享 Schema 校验最终响应，单个非法任务、无快照项目或不可用项目不能阻塞其他项目。

### 共享 API 合同

新增以下共享结构，继续放在 `src/shared/api.ts`：

```ts
TaskCollectionApi = "active" | "archived"

TaskCenterItemApi = {
  projectId: string
  collection: TaskCollectionApi
  task: TaskSummaryApi
  parentTitle: string | null
}

TaskCenterResponse = {
  projects: ProjectListItem[]
  tasks: TaskCenterItemApi[]
}
```

响应复用 `ProjectListItem`，避免重新定义焦点、历史、不可用、监听模式、快照时间和诊断数量语义。任务项只保存 `projectId`，前端通过项目索引关联展示信息，避免在每一行重复项目对象。

### 排序稳定性

服务端只提供稳定的原始聚合顺序，最终用户排序由前端完成。任务缺少 `updatedAt` 时排在有更新时间的任务之后；所有排序最终使用项目名称、任务标题、任务 ID 和 `sourcePath` 作为确定性回退键。

## 前端设计

### 全局导航

任务中心是全局页面，不加入单项目的“概览、Spec、Task、Workflow、诊断”标签。项目侧栏增加“任务中心”入口，位于实时连接状态与“添加项目”操作之间。

控制台增加顶层工作区模式：

- `project`：保持现有单项目工作区。
- `tasks`：展示跨项目任务中心。

选择任意侧栏项目时切回 `project` 模式；点击任务中心入口时保留最近选中的项目状态，但主区域切换到 `tasks` 模式。

### 状态与 Hook 边界

新增 `useTaskCenter` Hook，负责：

- 调用 `GET /api/tasks` 并管理独立加载、错误和后台刷新状态。
- 管理项目范围、任务集合、关键词、项目、状态、阶段、负责人、包和排序状态。
- 基于响应构建项目索引、筛选选项、当前结果、汇总数量和确定性排序结果。
- 在任务中心未激活时保留状态但不因普通项目视图变化重复请求。

`useProjectConsole` 继续负责项目列表、项目详情、单项目文档、顶层工作区模式、URL 协调和唯一 SSE 连接。它内部组合 `useTaskCenter`，避免创建第二条 EventSource 连接，也避免把全部筛选算法继续堆入现有 Hook。

### URL 状态

URL 使用顶层 `mode=tasks` 区分任务中心。任务中心参数如下：

| 参数 | 含义 | 默认值 |
| --- | --- | --- |
| `scope` | `focus` 或 `all` | `focus` |
| `collection` | `active`、`archived` 或 `all` | `active` |
| `q` | 关键词 | 空 |
| `taskProject` | 项目 ID | 全部 |
| `status` | 状态 | 全部 |
| `phase` | 阶段 | 全部 |
| `assignee` | 负责人 | 全部 |
| `package` | 包 | 全部 |
| `sort` | `updated_desc` 或 `updated_asc` | `updated_desc` |

将现有 URL 解析和写入逻辑扩展为项目模式与任务中心模式的判别联合。写入任务中心 URL 时不混入 `spec`、`task`、`document` 等项目阅读参数；返回项目模式时恢复项目自身的稳定选择。

URL 恢复分为两步：同步解析时，非法 `scope`、`collection`、`sort` 回退表格中的默认值；任务中心响应到达后，再校验 `taskProject`、`status`、`phase`、`assignee` 和 `package` 是否仍存在于当前范围与集合的可选值中，不存在则清除。任何修复都通过 `history.replaceState` 将有效状态重新写回 URL，避免刷新后重复恢复非法值，也不新增浏览器历史记录。

### 搜索与筛选

过滤顺序固定为：项目范围 → 任务集合 → 项目 → 状态 → 阶段 → 负责人 → 包 → 关键词。

关键词使用 `trim().toLowerCase()` 归一化，对项目名称、任务标题、任务 ID、状态、阶段、负责人和包执行包含匹配。状态搜索语料同时包含原始值和 `formatTaskStatus` 生成的中文展示值；空值字段按空字符串参与匹配，不产生异常。

状态筛选和汇总复用现有兼容语义。内部状态分组固定为：

| 分组 | 原始状态 | 页面语义 |
| --- | --- | --- |
| `planning` | `planning` | 规划中 |
| `in_progress` | `in_progress` | 实施中 |
| `review` | `review` | 评审中 |
| `completed` | `completed`、`done` | 已完成 |
| `other` | 其他非空原始状态 | 其他 |

状态筛选项仍按原始状态展示未知值，`done` 与 `completed` 共用“已完成”筛选语义；状态汇总只展示上述五组，未知状态统一计入“其他”。关键词搜索语料同时包含原始状态和 `formatTaskStatus` 的中文展示值。五组状态数量之和必须与当前结果总数一致。

每个筛选维度首版只允许选择一个值。项目选项来自当前项目范围，状态、阶段、负责人和包选项来自当前项目范围与任务集合内的任务，但不受关键词和同级属性筛选影响。切换范围或集合后，已经不存在的筛选值自动清除。

汇总数量基于全部已选范围、筛选条件和关键词后的结果计算，包括总数、活动、归档、四类标准状态和“其他”数量，必须与列表结果一致。

### 排序规则

在 `scope=all` 时先按项目状态排序：焦点项目在历史项目之前。随后应用更新时间升序或降序。更新时间相同或为空时，依次使用项目名称、任务标题、任务 ID 和 `sourcePath` 排序。

### 页面布局

任务中心使用信息密集的全宽工作布局，不使用嵌套卡片：

1. 顶部标题和当前项目/任务汇总。
2. “焦点 / 全部”和“活动 / 归档 / 全部”分段控件。
3. 搜索框、项目、状态、阶段、负责人、包和排序菜单，以及清除条件命令。
4. 紧凑汇总行。
5. 扁平任务列表。

桌面列表稳定展示项目、任务、状态、阶段、负责人、包和更新时间。子任务在标题下显示父任务名称。历史项目行显示“历史快照”和最后索引时间。窄屏改为两行紧凑布局，保留项目、标题、状态和更新时间，其他元数据自然换行，不能产生页面级水平滚动。

### 任务跳转

点击焦点项目任务时，直接切换到现有项目 Task 页面并选中对应 `sourcePath`。

点击历史项目任务时，将该点击视为用户对源项目的显式访问：

1. 先调用现有项目刷新接口，使当前进程获得正文读取资格。
2. 刷新成功后进入 Task 页面并选中任务。
3. 如果刷新后任务已不存在，进入该项目 Task 页面并提示“任务快照已更新，原任务已不存在”。
4. 如果项目变为不可用，进入项目诊断视图并展示现有错误。

该流程不自动把历史项目加入焦点，也不修改任何源项目文件。

现有 `readProjectTaskDetail()` 已按 `prd.md`、`design.md`、`implement.md`、其余 `relativePath` 建立稳定文档顺序，前端不重复实现第二套排序，而是从最新任务详情响应中选择首个文档。选择结果和 `project`、`task` 一起写入最终 URL；无文档时 `document` 保持 `null` 且不写入 URL。历史项目必须在显式刷新并取得最新任务详情后重新选择，不能沿用聚合快照中的旧文档状态。

## SSE 与数据一致性

应用继续只维护一条 `/api/events` 连接。`project-invalidated` 与 `project-reindexed` 可能在一次手动刷新中连续到达，因此事件处理使用约 150ms 的尾随防抖合并同一批变化。每批有效事件只执行一次：

- 后台刷新项目列表。
- 增加任务中心刷新代次；任务中心激活时重新请求 `GET /api/tasks`。
- 事件属于当前选中项目时，继续刷新现有项目详情。

若一批事件涉及多个项目，任务中心仍只刷新一次，当前项目详情仅在该批包含当前项目时刷新一次。SSE 回调通过当前项目 Ref 读取选择，切换项目或任务中心模式不应重建 EventSource。请求提交继续使用项目 ID、当前选择和请求代次保护，避免旧响应覆盖新页面；组件卸载时清理防抖计时器。

## 错误与空状态

- 没有已登记项目：保持现有项目发现入口，不显示无意义的任务筛选控件。
- 没有焦点项目：任务中心显示焦点为空，并提供切换到全部项目的明确入口。
- 当前条件无结果：保留条件，展示一键清除入口。
- 聚合 API 失败：只影响任务中心，不覆盖侧栏和现有项目详情。
- 项目无快照：不产生任务行，在汇总说明中计数。
- 项目不可用：不产生任务行，在汇总区域提供进入诊断的入口。
- 未知状态、阶段、负责人或包：保持可读原值，不破坏筛选和布局。

## 性能与安全

- 聚合只读取应用内注册表和已有快照，不访问源项目正文。
- 搜索和筛选在内存中执行，不产生每次输入的网络请求。
- 不新增数据库、索引文件、Worker、WebSocket 或远程服务。
- API 继续使用项目 ID 和现有共享 Schema，不接受任意文件路径。
- 不增加服务端分页或持久化索引，任务中心仍保留完整元数据，以保证本地搜索、筛选和汇总语义一致。
- `useTaskCenter` 以 `refreshGeneration + retryGeneration` 作为请求缓存键；已有成功响应且缓存键未变化时，任务中心重新激活只复用内存响应，SSE 失效或显式重试才重新请求。
- 关键词输入使用约 200ms 延迟值驱动筛选和排序，URL 与输入状态保持即时更新。
- 页面默认只渲染排序结果前 100 条，“加载更多”每次追加 100 条；汇总仍基于完整结果，筛选或响应变化时重置可见数量。
- 若真实基准证明全量响应解析或内存本身成为瓶颈，再单独设计服务端分页、增量索引或虚拟列表，不在首版提前引入。

## 验证策略

按项目约定不新增测试类。使用临时多项目 fixture 和 Playwright 验证：

- 焦点、历史、不可用、无快照项目同时存在时的聚合结果。
- fixture 中包含非法 `task.json`、缺失父任务引用和循环父子关系，断言损坏任务被诊断或降级且不会阻塞其他项目任务。
- 默认焦点/活动范围，以及项目范围和集合切换。
- 项目、状态、阶段、负责人、包的组合筛选和元数据关键词搜索。
- 更新时间升降序、空时间和稳定回退排序。
- 汇总数量与当前结果一致。
- URL 刷新恢复全部任务中心状态，并验证非法枚举值和过期筛选值会被清理后写回 URL。
- 焦点任务直接跳转、历史任务刷新后跳转、任务消失和项目不可用分支；断言最终 URL 包含正确的 `project`、`task` 和存在时的默认 `document`。
- 连续发送 `project-invalidated` 与 `project-reindexed`，断言约 150ms 合并窗口内项目列表和任务中心各只刷新一次，且连接不因视图切换重建。
- 375px、768px、1024px、1440px 无页面级水平滚动、文字遮挡或控制错位。

## 回滚与兼容

本功能只增加共享 DTO、只读 API、前端页面与 URL 参数，不修改注册表或快照持久化 Schema，无数据迁移。回滚时删除任务中心路由、响应合同、Hook、组件和样式，并移除全局导航入口即可；现有项目 API、单项目页面和应用数据保持兼容。
