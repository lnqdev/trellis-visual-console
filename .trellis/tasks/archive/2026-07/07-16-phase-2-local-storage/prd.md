# 阶段二本机注册表与摘要快照

## 目标

建立应用自有的本机持久化层，可靠保存项目注册信息和可重建摘要快照，为后续扫描、解析、焦点监听和只读 API 提供稳定数据合同，同时保证任何存储操作都不会写入被浏览项目。

## 背景与已确认事实

- 源项目 `.trellis/` 是唯一事实来源，注册表与快照只是应用自有配置和缓存。
- 首版使用版本化 JSON 文件，不引入数据库服务器。
- 文件保存在操作系统应用数据目录，而不是任意项目仓库。
- 注册表保存项目身份、路径、显示状态和索引状态；快照保存历史项目可展示的摘要。
- 本阶段不负责发现项目、解析 Trellis 文件或提供正式 HTTP API。

## 需求

1. 定义版本为 `1` 的项目注册表结构，包含稳定 ID、项目路径、显示名称、`history | focus | unavailable` 状态、最近访问时间、最后索引时间和错误信息。
2. 定义版本为 `1` 的摘要快照结构，包含项目概览、Spec 树、活动/归档 Task 摘要、Workflow 摘要、索引时间和诊断信息。
3. macOS、Windows、Linux 默认使用各自应用数据目录，并允许通过 `TRELLIS_VISUAL_CONSOLE_DATA_DIR` 覆盖目录用于开发和验证。
4. 注册表写入 `registry.json`，摘要快照写入 `snapshots.json`，两者不写入源项目。
5. JSON 写入采用同目录临时文件、落盘同步和原子重命名，避免进程中断留下半文件。
6. 同一文件的并发写入在进程内串行化，避免较早操作覆盖较新操作。
7. 缺失文件自动创建默认结构；JSON 损坏或结构非法时，将原文件隔离为带时间戳的 `.corrupt-*` 文件并恢复默认结构。
8. 数据版本与当前版本不一致时拒绝覆盖，保留原文件供未来迁移。
9. 服务启动时初始化应用存储，并记录发生的损坏恢复；不增加注册表或快照 HTTP API。
10. 所有新增方法添加中文注释，关键恢复和原子写流程添加中文说明。
11. 本阶段不生成测试文件。

## 验收标准

- [x] 首次初始化可创建应用数据目录、`registry.json` 和 `snapshots.json`。
- [x] 注册表数据保存后可重新加载，字段和值保持一致。
- [x] 摘要快照保存后可重新加载，项目概览、Spec、Task、Workflow 和诊断数据保持一致。
- [x] 损坏 JSON 被隔离，原路径恢复为合法的版本 1 默认文件。
- [x] 不兼容版本抛出明确错误，原文件内容保持不变。
- [x] 多次并发保存最终得到最后一次调用的数据，文件始终是合法 JSON。
- [x] 删除应用数据目录不会删除或修改模拟源项目中的任何文件。
- [x] 服务仍只绑定 `127.0.0.1`，健康接口和阶段一生产静态托管保持正常。
- [x] `pnpm lint`、`pnpm typecheck`、`pnpm build` 全部通过。

## 不在本阶段范围内

- 扫描目录、手动添加项目和 `.trellis/` 有效性校验。
- 从真实 Trellis 文件生成摘要快照。
- 焦点项目重新索引、监听器和 SSE。
- 注册表、快照或项目内容的 HTTP API 与 Web UI。
- 数据库、云同步、跨机器共享和 Core SDK。
