# 桌面端支持在线更新

## 目标与用户价值

让已安装的 Trellis Visual Console 能安全发现并安装新版本，用户无需手动查找、下载和覆盖安装包，同时保持现有本机只读控制台的安全、隐私与跨平台交付边界。

## 已确认事实

- 当前应用基于 Tauri 2，支持 macOS 13+（arm64、x64 DMG）和 Windows 10 22H2/11 x64（当前用户 NSIS），产品标识为 `com.wanglinqiao.trellis-visual-console`。证据：`src-tauri/tauri.conf.json:2-4, 31-43`、`README.md:94-112`。
- 当前版本为 `0.1.0`，尚未配置 Tauri Updater 插件、更新公钥、更新端点或发布流水线。证据：`package.json:3`、`src-tauri/Cargo.toml:19-24`、`src-tauri/tauri.conf.json:2-46`。
- 现有生产运行合同禁止 Node、本地 HTTP 服务、浏览器、sidecar、托盘和第二窗口；在线更新需要把受控 HTTPS 请求补充为唯一明确的运行期联网例外。证据：`.trellis/spec/backend/desktop-runtime-contract.md:11-21`。
- 首版桌面客户端明确将自动更新、Apple Developer ID 公证、Windows 商业代码签名和面向公众的自动发布流水线排除在范围外；本任务需要重新定义这些边界。证据：`.trellis/tasks/archive/2026-07/07-20-desktop-client/prd.md` 的“Out of Scope”、`README.md:137-141`。
- macOS 当前仅交付未签名内测 DMG；公开分发仍需补齐签名与公证。证据：`README.md:115-128`。
- 应用不采集遥测，且此前运行期不主动访问网络；在线更新会成为必须明确告知并可控的唯一例外。证据：`README.md:145-158`。

## 需求

- **R1**：提供签名验证的更新发现、下载、安装和重启流程，拒绝无效、篡改、降级或与当前平台/架构不匹配的更新。
- **R2**：更新检查仅连接明确配置的 HTTPS 发布端点；不得引入本地 HTTP 服务、Node/Go sidecar、遥测或无关联网行为。
- **R3**：用户能理解更新状态、失败原因和下一步操作，并能在更新安装前作出明确选择；主界面全局区域与诊断页均须显示当前应用版本，便于重启后核对更新结果。
- **R4**：发布流程能为支持的平台和架构生成可验证的更新产物及对应元数据；私钥通过安全本机存储或未来的 CI Secret 在构建时注入，不进入仓库或安装包。
- **R5**：更新失败、取消或重启失败不得破坏现有安装、应用数据或已登记的 Trellis 项目。
- **R6**：首个在线更新版本同时支持现有全部交付目标：macOS arm64、macOS x64 和 Windows x64；完成一次性过渡安装后，不得让任一现有平台继续依赖手工覆盖安装。
- **R7**：第一版使用公开 Gitee Release 免费托管三个目标的更新安装包，并通过公开 Gitee 仓库地址提供 `latest.json`；首版允许手动发布，设计须保留后续迁移到对象存储/CDN 和自动发布流水线的能力。
- **R8**：应用在窗口可操作后后台检查更新，24 小时内最多自动检查一次；自动检查只读取 `latest.json`，不自动下载或安装。发现更新后由用户明确确认，诊断页始终提供手动检查入口。
- **R9**：首个版本按免费受控内测交付。所有更新均须通过 Tauri 更新签名校验；macOS 不将 Apple Developer ID 签名或公证作为前置条件，Windows 不将商业代码签名作为前置条件，但界面和发布说明必须清楚标明内测版及 Gatekeeper/SmartScreen 提示。
- **R10**：发现更新时必须展示目标版本和中文更新说明，用户确认下载前可阅读。更新说明复用对应 Gitee Release 的发布内容，并写入更新清单。
- **R11**：用户确认后执行 Tauri 的下载并安装流程。macOS 安装完成后提供“立即重启”和“稍后重启”；选择稍后重启仅保留当前旧进程，应用包已更新，下次正常启动直接运行新版本且不得重复下载。Windows 安装阶段须先明确提示应用即将关闭，确认后由安装器完成更新。
- **R12**：当前 `0.1.0` 客户端不含更新器，首个过渡版本固定为 `0.2.0-beta.1`，所有现有用户按其平台手工安装一次；此后的较高版本通过应用内更新交付，发布页面必须说明该一次性迁移步骤。
- **R13**：每次发布必须同时具备 macOS arm64、macOS x64、Windows x64 的同一版本更新产物、签名、中文更新说明和更新清单；任一目标缺失时不得更新公开 `latest.json`。
- **R14**：出现严重发布问题时，先冻结或撤下 `latest.json`，阻止更多客户端发现该更新；已升级用户只通过版本号更高的修复版本恢复，不支持静默降级或自动回退。

## 验收标准

- [ ] **AC1（R1）**：受支持平台和架构的已安装应用可发现比当前版本新的已签名发布，完成下载、安装和重启后运行目标版本；无效签名、错误架构、相同/更低版本均不会替换当前安装。
- [ ] **AC2（R2）**：运行期网络访问仅发生在用户确认的更新检查、下载或明确配置的自动检查中，目标仅为批准的 HTTPS 发布端点；生产进程不启动本地服务或 sidecar。
- [ ] **AC3（R3）**：界面覆盖无更新、发现更新、下载中、下载失败、签名/元数据校验失败、安装失败、需要重启和用户取消状态，文案为中文且可恢复；任意主界面状态都能看到当前版本，诊断页可同时核对版本与更新状态。
- [ ] **AC4（R4）**：发布产物包含版本、目标平台/架构、下载地址、签名与校验信息；签名私钥只从安全构建环境注入，仓库只保存公开验证材料和非敏感配置。
- [ ] **AC5（R5）**：失败、取消和异常退出后，原安装可启动，固定应用数据目录和已登记项目保持不变；更新前后遵守现有单实例与关闭清理合同。
- [ ] **AC6（R6）**：macOS arm64、macOS x64 和 Windows x64 均从已安装的 `0.2.0-beta.1` 完成真实在线升级验收，升级后平台、架构、版本及核心功能正确。
- [ ] **AC7（R7）**：无需登录或访问令牌的已安装客户端可从公开 Gitee 地址读取 `latest.json` 并下载对应更新包；更换发布基地址不要求重写更新业务流程或变更签名信任根。
- [ ] **AC8（R8）**：首次启动可后台检查，连续重启在 24 小时内不重复请求；手动检查不受该限制，自动检查和手动检查均不下载更新，直到用户在界面明确确认。
- [ ] **AC9（R9）**：有效 Tauri 更新签名的内测包可以升级；无效签名一律拒绝。macOS 与 Windows 的内测安装/系统拦截说明在应用和发布材料中均可见，且不要求配置 Apple 或 Windows 商业签名凭据。
- [ ] **AC10（R10）**：每个可用更新在确认下载前均展示版本号与非空中文更新说明；缺少说明的发布清单不可作为可安装更新提供。
- [ ] **AC11（R11）**：macOS 安装完成后选择稍后重启，再次启动直接运行新版本且更新端点没有第二次下载请求；选择立即重启运行新版本。Windows 在用户确认安装后退出应用并由安装器完成更新。
- [ ] **AC12（R12）**：`0.2.0-beta.1` 的三个目标手工安装包和迁移说明可获取；从该版本升级到更高版本的三平台流程均无需再次手工下载安装包。
- [ ] **AC13（R13）**：任一正式更新清单的三个目标均指向相同 SemVer 版本且各自拥有有效签名与下载地址；缺少任一目标的候选发布不会被公开客户端发现。
- [ ] **AC14（R14）**：演练冻结更新清单后，旧客户端不再发现问题版本；发布更高修复版本后，已升级客户端可发现并完成修复升级，且不存在向低版本回退的路径。

## 范围外

- 远程访问、账号、云同步、遥测、A/B 灰度和自动化数据收集。
- 为 Linux、Windows ARM64 或 macOS Universal 新增安装包。
- 以更新功能重构现有项目浏览、存储或实时刷新业务。
- Apple Developer ID、公证、Windows 商业代码签名、应用商店分发和面向陌生公众的无拦截安装体验。
- 自动发布流水线、灰度通道、静默下载安装、自动降级和自动回退。
